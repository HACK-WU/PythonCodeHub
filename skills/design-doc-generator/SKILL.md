# 设计文档生成器

## 何时使用

仅在以下情况使用本 skill：

- 用户已经能给出**一段较完整的需求描述**（不是一句话标题）
- 用户的目标是产出一份**给小团队评审用的轻量级设计文档**
- 用户希望**结构完整但不啰嗦**

若只是粗略想法、一句话标题、或仅需问答澄清需求，**不要使用本 skill**，先帮用户把需求说清楚。

### 轻量级版本快速通道

当需求非常简单（如仅涉及单文件修改、逻辑清晰无歧义）时，可跳过阶段 2（语义分析）、阶段 2.5（需求拆分）和阶段 3（大纲），直接用精简模板输出设计文档。精简模板保留第 1、4、5、10、14 章共 5 个核心章节。

判断是否走快速通道的标准：
- 需求只涉及 **1~2 个文件**的改动
- 需求描述中**无歧义词**
- 方案只有**一条明确路径**，无需多方案对比

快速通道仍需阶段 1 的需求摘要确认。

## 核心原则

1. **语义先行**：在写任何设计内容前，先与用户对齐需求语义和适用场景
2. **轻量但完整**：14 章骨架全部保留，但每章只写关键点，不堆套话
3. **必须有"不在范围内"**：每份文档都要明确边界，避免范围蔓延
4. **图表优先**：能用 mermaid 表达的关系，不用文字硬描述
5. **分阶段确认**：每个关键阶段必须等用户确认才能继续
6. **不脑补需求**：发现需求模糊或场景不成立时，停下来对齐，而非自行补全

## 工作流总览

```
阶段 1：需求摘要确认       → 输出"我理解的需求"，等用户确认
阶段 2：语义分析与场景对齐 → 拆解语义、匹配场景、判定价值，等用户确认
阶段 2.5：需求拆分（条件）→ 复杂需求拆为子需求 + 依赖 DAG，等用户确认；简单需求跳过
阶段 3：大纲与边界输出     → 拆分需求：父文档大纲 + 各子文档大纲 + 不在范围内清单
                             未拆分需求：14 章骨架 + 不在范围内清单
                             等用户确认
阶段 4：分章节填充         → 拆分需求：先填充父文档，再按依赖拓扑序填充各子文档
                             未拆分需求：按确认后的大纲填充
阶段 5：落盘归档           → 写入文件，确认路径；质量自检（拆分场景含一致性校验）
阶段 6：（可选）实施计划   → 是否再产出一份 IMPL_PLAN
```

**未得到用户对当前阶段的确认前，不进入下一阶段。**

### 需求拆分说明

阶段 2 完成后，进入阶段 2.5 判定需求复杂度。满足拆分条件时，将需求拆为多个子需求。拆分后的文档结构为：

- **1 份父文档**：覆盖全局架构、模块边界、接口契约、全局术语、全局风险（精简模板，约 6 章）
- **N 份子文档**：各自按完整 14 章模板编写详细设计，范围为对应子需求

父文档必须先于子文档完成填充和落盘，子文档按依赖 DAG 的拓扑序依次编写。子文档中引用的接口签名、术语定义必须与父文档保持一致。

### 回退与迭代机制

用户可在任意阶段要求回退到之前的阶段：
- **回到阶段 1**：用户发现需求本身有误，需要重新定义
- **回到阶段 2**：用户在阶段 2.5/3/4 发现语义理解有偏差，需要重新对齐
- **回到阶段 2.5**：用户在阶段 3/4 发现拆分粒度不合理，需要重新拆分
- **回到阶段 3**：用户在阶段 4 填充时想调整章节结构

回退时，丢弃回退点之后的所有产出，从该阶段重新开始。不需要重做回退点之前的阶段（除非用户要求）。

### 与其他 Skill 的协作

本 skill 可与以下 skill 配合使用：

| 协作 Skill | 协作方式 | 触发场景 |
| --- | --- | --- |
| `create-skill` | 设计文档可作为新 skill 的设计输入；设计文档中的关键决策应写入新 skill 的 SKILL.md | 用户要求基于设计文档创建新 skill |
| `shared-memory` | 设计过程中的关键架构决策、技术选型结论应存入共享记忆，供后续会话直接引用 | 设计完成后主动保存；或后续对话中检测到重复探索相同架构问题时触发 |

协作触发方式：在阶段 5 落盘完成后，询问用户是否需要：
1. 基于设计文档创建新 skill（调用 `create-skill`）
2. 将关键决策存入共享记忆（调用 `shared-memory`）

## 阶段 1：需求摘要确认

将用户给出的需求复述为**结构化摘要**，逼出隐含假设。

输出格式：

```text
📌 需求摘要
━━━━━━━━━━━━━━━━
原始描述：<原文一句话回放>
核心诉求：<去掉修饰后的本质诉求，一句话>
关键动词：<识别出"加一个"、"重构"、"替换"、"扩展"等动作类型>
触发主体：<谁会用 / 谁会触发>
预期产物：<用户希望最终得到什么形态的产物>

请确认以上理解是否正确，或补充我遗漏的关键点。
```

用户确认后进入阶段 2。

## 阶段 2：语义分析与场景对齐（关键步骤）

**这是本 skill 最重要的步骤，不可跳过、不可合并到其他阶段。**

### 这一步要解决的核心问题

> 同一句需求描述，不同人理解不同 → 设计方向跑偏 → 文档作废。

### 三个动作

#### ① 语义拆解

把需求（含子需求）逐条提取，列出每条的**精确重述**——用更明确的术语替换可能存在歧义的词。

常见歧义词举例（不限于此）：
- "幂等" → 接口幂等 / 任务幂等 / 消息幂等
- "缓存" → 进程内 / 分布式 / 浏览器侧 / CDN
- "异步" → 线程异步 / 协程 / 消息队列 / 定时任务
- "重试" → 客户端重试 / 服务端补偿 / 中间件重投
- "限流" → 单机 / 全局 / 用户维度 / 接口维度
- "通知" → 推送 / 邮件 / 站内信 / Webhook

#### ② 场景匹配

针对每条子需求，明确：
- **触发场景**：什么情况下会用到
- **使用主体**：谁来调用 / 谁来感知
- **解决痛点**：当前没有它会怎样

#### ③ 价值判定

每条子需求三选一：

| 标记 | 含义 | 处理方式 |
| --- | --- | --- |
| ✅ 匹配 | 场景真实、价值成立 | 进入设计 |
| ⚠️ 存疑 | 场景模糊或价值不明 | 提出具体反问，等用户回答 |
| ❌ 不匹配 | 场景不成立、需求无意义、有更优方案 | **主动劝退或推荐替代方案，不进入设计** |

### 分级追问机制

当需求存在**多维度歧义**时（例如同时有"含义歧义"和"粒度歧义"），不要一次性列出所有排列组合让用户回答，而是**逐维度追问**：

1. **第一轮**：识别出所有歧义维度，按影响程度排序，先针对**最关键的维度**提问（二选一）
2. **第二轮**：基于用户第一轮的回答，再对次关键维度提问
3. **循环**：直到所有维度都有明确答案

每轮反问格式：
```text
该需求存在 N 个歧义维度，我将逐个与您确认：

🔴 维度 1（最关键）：<歧义描述>
→ 请选择：A.<含义1> / B.<含义2>

其余维度将在确认此维度后逐一确认。
```

**禁止**：一次性抛出所有维度的排列组合反问（用户难以一次性回答）。

### 输出格式

```text
🧠 语义分析与场景对齐
━━━━━━━━━━━━━━━━
原始需求：<回放>

【语义拆解】
- 词项 A 在此处可能指：
  - 含义 1：...
  - 含义 2：...
  根据上下文推断：含义 X
- 词项 B ...

【场景匹配】
| 子需求 | 触发场景 | 使用主体 | 解决痛点 |
| ----- | ------- | ------- | ------- |
| ...   | ...     | ...     | ...     |

【价值判定】
- ✅ 子需求 1：...（匹配，进入设计）
- ⚠️ 子需求 2：...（存疑，请回答：<具体反问>）
- ❌ 子需求 3：...（不匹配，建议改用：<替代方案>，理由：<...>）

请确认以上判定，或纠正其中的语义/场景理解。
```

### 关键约束

- **必须停下来等用户确认**，不能自己拍板继续
- 出现 ❌ 时，**不能为了"完成任务"硬写设计**，要先劝退
- 出现 ⚠️ 时，反问必须**具体到二选一/三选一**，不能问"你能再说一下吗"
- 多维度歧义时，使用**分级追问**逐维度确认，不一次性列出所有组合
- 用户回答"按你的判断"时，明确告知你将采纳的语义解释，再进入下一阶段
- 详见 [reference.md](reference.md) 中的语义分析示例

## 阶段 2.5：需求拆分（条件触发）

### 触发条件

阶段 2 完成后，按以下维度判定是否需要拆分：

| 判定维度 | 不拆 | 拆 |
|----------|------|-----|
| 子需求数量 | ≤ 3 个 | > 3 个 |
| 子需求间耦合度 | 高耦合（共享核心模块） | 低耦合（边界清晰、接口明确） |
| 设计决策交叉度 | 一个决策影响多个子需求 | 子需求可独立决策 |
| 单文档预估篇幅 | 14 章 × 200 字可覆盖 | 关键章节超 200 字上限 |

**核心判据**：耦合度 > 数量。2~3 个高耦合子需求不拆（单文档内分节处理）；4 个以上或低耦合子需求拆分。

进入拆分流程需同时满足：
1. 子需求数量 ≥ 2 且耦合度为低（边界清晰、可独立决策）
2. 子需求数量 > 3 时，无论耦合度高低均建议拆分（数量本身增加管理复杂度）

**不拆分但需结构化处理的场景**：2~3 个子需求且耦合度高 → 不拆文件，但在单份文档内按子需求分节，便于分批评审。

### 拆分动作

1. **识别子需求**：将需求拆为 N 个边界清晰的子需求，每个子需求需满足：
   - 有独立的触发场景和使用主体
   - 可独立定义接口契约
   - 可独立验证

2. **构建依赖 DAG**：识别子需求间的依赖关系，用 mermaid DAG 图表示

3. **定义文档结构**：
   - 父文档（1 份）：覆盖全局视角，章节范围固定为 → 需求背景 & 目标、名词术语表、方案设计（架构层）、架构图/流程图、模块/类设计（边界与契约）、风险 & 待定问题
   - 子文档（N 份）：各自覆盖对应子需求的完整 14 章详细设计

### 输出格式

```text
🔀 需求拆分
━━━━━━━━━━━━━━━━
原始需求：<回放>

【复杂度判定】
- 子需求数量：N 个
- 耦合度：高 / 低
- 决策交叉度：高 / 低
- 结论：✅ 需要拆分 / ⏭ 跳过拆分

【子需求清单】
| 编号 | 子需求 | 触发场景 | 使用主体 | 独立验证方式 |
| ---- | ------ | -------- | -------- | ------------ |
| S-01 | ...    | ...      | ...      | ...          |
| S-02 | ...    | ...      | ...      | ...          |

【依赖关系】
    flowchart LR
        S-01 --> S-02
        S-01 --> S-03
        S-02 --> S-04

（上述为 mermaid 语法示意，实际输出时用 mermaid 代码块渲染）

依赖说明：
- S-02 依赖 S-01（<原因>）
- S-03 依赖 S-01（<原因>）
- S-04 依赖 S-02（<原因>）

【文档规划】
- 父文档：<feature>_DESIGN.md（全局架构 + 模块边界 + 接口契约）
- 子文档：S-01 → <feature>_S01_<子需求名>_DESIGN.md
         S-02 → <feature>_S02_<子需求名>_DESIGN.md
         ...

请确认拆分方案，或调整子需求粒度 / 依赖关系。
```

**结论为"跳过拆分"时**，直接进入阶段 3，输出格式简化为：

```text
🔀 需求拆分
━━━━━━━━━━━━━━━━
复杂度判定：子需求 N 个，耦合度 高，无需拆分。
直接进入阶段 3。
```

### 关键约束

- **必须停下来等用户确认**，不能自行决定拆分后跳过
- 用户可调整：子需求粒度（合并或进一步拆分）、依赖关系、文档命名
- 拆分后阶段 3 的大纲需覆盖父文档 + 所有子文档
- 父文档模板见 [PARENT_TEMPLATE.md](PARENT_TEMPLATE.md)

## 阶段 3：大纲与边界输出

### 未拆分需求

按 [TEMPLATE.md](TEMPLATE.md) 的 14 章骨架，给出**章节级大纲**（不是完整内容），并附上**不在范围内**清单。

输出格式：

```text
📋 设计文档大纲
━━━━━━━━━━━━━━━━
建议文档名：<feature>_DESIGN.md
建议路径：<相关模块>/<feature>_DESIGN.md

章节安排：
1. 需求背景 & 目标               — 1 段背景 + 3 条目标
2. 名词术语表                    — N 个易混词
3. 现状分析（AS-IS）             — 当前实现痛点
4. 方案设计（TO-BE）             — 核心方案 1 句话 + 关键决策点
5. 架构图 / 流程图               — mermaid 流程图 1 张
6. 模块/类设计                   — N 个核心类/模块
7. 接口设计                      — 公开 API 签名
8. 数据模型                      — 表结构 / DTO（无则标"不涉及"）
9. 关键流程时序图                — mermaid 时序图 1 张
10. 异常处理 & 边界情况          — 关键异常路径
11. 性能 & 安全考虑              — 仅写真实存在的考量
12. 测试方案                     — 单测 / 集成测试覆盖范围
13. 实施计划 / 里程碑            — 批次划分（可对接 IMPL_PLAN 风格）
14. 风险 & 待定问题              — 已知风险与开放问题

⛔ 明确不在本设计范围内：
- <项 1，例如：不涉及鉴权改造>
- <项 2>
- <项 3>

请确认大纲与边界，或调整章节增减、合并、命名。
```

### 拆分需求

分别给出父文档和各子文档的大纲。父文档使用 [PARENT_TEMPLATE.md](PARENT_TEMPLATE.md) 的精简骨架；子文档使用 [TEMPLATE.md](TEMPLATE.md) 的 14 章骨架。

输出格式：

```text
📋 设计文档大纲（拆分）
━━━━━━━━━━━━━━━━

【父文档】<feature>_DESIGN.md
章节安排：
1. 需求背景 & 目标               — 全局背景 + 整体目标 + 不在范围内
2. 名词术语表                    — 全局共享术语
3. 方案设计（TO-BE）             — 整体架构 + 子需求划分 + 关键决策点
4. 架构图 / 流程图               — 全局架构 mermaid 图 1 张
5. 模块/类设计                   — 子需求边界 + 接口契约
6. 风险 & 待定问题              — 全局风险与跨子需求依赖风险

⛔ 明确不在本设计范围内：
- ...

【子文档 S-01】<feature>_S01_<子需求名>_DESIGN.md
章节安排：
1. 需求背景 & 目标               — 本子需求的背景和目标
2. 名词术语表                    — 本子需求特有术语（全局术语见父文档）
3. 现状分析（AS-IS）             — ...
（以下按 14 章骨架列出，格式同未拆分场景）
⛔ 本子需求不在范围内项

【子文档 S-02】<feature>_S02_<子需求名>_DESIGN.md
（同上格式）

...
⛔ 各文档共同不在范围内：
- <全局排除项>

请确认大纲与边界，或调整章节增减、合并、命名。
```

### 章节可省略规则（通用）：
- 第 8 章无数据持久化时写"不涉及"，不删除
- 第 9 章简单同步流程可写"无关键时序"，不删除
- 第 11 章无明显性能/安全压力时写"无特殊考虑"，不删除
- 其余章节**不可省略**

## 阶段 4：分章节填充

### 未拆分需求

确认大纲后，按章节顺序填充。每章遵循以下风格约束：

- **每章 ≤ 200 字正文**（图表不计），超过先反思是否在堆套话
- 列表项 ≤ 5 条，超过应分子节
- 每章可有 0~1 张图（mermaid 优先）
- 不写"众所周知"、"显而易见"、"业界常用"等空话
- 不复述阶段 1/2 已确认的内容

填充顺序建议：1 → 2 → 4 → 5 → 6 → 7 → 8 → 9 → 3 → 10 → 11 → 12 → 13 → 14

> 解释：先输出"目标 + 方案主干 + 架构 + 模块 + 接口"这条主线，再回头补"现状 / 异常 / 性能 / 测试 / 实施 / 风险"，避免在主干未定时被周边章节带偏。

### 拆分需求

分两轮填充：

**第一轮：填充父文档。** 按父文档 6 章骨架顺序填充，重点关注：
- 第 3 章「方案设计」必须明确各子需求的职责边界和协作接口
- 第 5 章「模块/类设计」必须给出跨子需求的接口契约（函数签名级别）
- 父文档同样遵循 ≤ 200 字/章 的风格约束

**第二轮：按依赖拓扑序填充各子文档。**
- 无依赖的子需求对应的子文档可按任意顺序填充
- 有依赖的子需求必须在所依赖的子文档完成后填充
- 子文档填充时遵循与未拆分需求相同的风格约束和填充顺序
- 子文档中引用的接口签名、术语必须与父文档保持一致

填充时的写作要点详见 [reference.md](reference.md) 的"章节写作要点"小节。

## 阶段 6（可选）：实施计划

若用户需要进一步落地，按 `COMMON_CELERY_IMPL_PLAN.md` 风格生成 `<feature>_IMPL_PLAN.md`，包含：

- 批次划分总览表
- 每批次：范围 / 产出物 / 依赖 / 风险 / 验收标准 五元组
- 跨批次约定（测试 / 文档 / 兼容性 / 依赖 / 提交）
- 风险与预案

实施计划仅在用户明确要求时生成，不主动生成。

## 反模式（不要做）

- ❌ 跳过阶段 2 直接出大纲：会导致设计跑偏
- ❌ 用户没确认就连续输出多个阶段产物：浪费 token，无法纠偏
- ❌ 章节内容堆砌（"为了让本章节看起来饱满"）：违反轻量原则
- ❌ 把"不在范围内"写成空话（"不涉及不相关的事情"）：必须具体到点
- ❌ 发现 ❌ 子需求时仍硬写设计：违反劝退义务
- ❌ 把语义分析阶段和大纲阶段合并到一次输出：用户无法在中间纠偏
- ❌ 多维度歧义时一次性列出所有排列组合反问：用户难以一次性回答
- ❌ 回退时保留旧产出：必须丢弃回退点之后的所有产出
- ❌ 拆分时只看子需求数量不看耦合度：高耦合需求强行拆分会增加协调成本
- ❌ 子文档接口签名与父文档不一致：父文档的契约是子文档的围栏
- ❌ 子文档间重复写全局术语表和背景：子文档引用父文档即可，避免重复
