## 背景

当前 `project-memory` 的设计，如果直接把“存储结构”理解成“AI 最终会看到的结构”，很容易走向两个问题：

- 为了让 AI 看起来方便理解，底层结构不断膨胀，出现越来越重的嵌套对象
- 为了便于查询，把大量说明性字段塞进 `related_modules` 子项，导致冗余、难维护、难扩展

实际上，这里需要先明确一个前提：

### **存储结构，不等于 AI 最终看到的结果**

系统中间至少还会经过两步：

1. **查询层转换**：根据当前问题，把底层存储转换成适合当前问题的结果视图
2. **AI 视图裁剪**：把查询结果进一步压缩成少量、高相关、可解释的数据，再提供给 AI

因此，这份方案不再把重点放在“如何把所有信息都直接塞进一个层层嵌套的结构里”，而是改为：

### **relation-first + 归一化存储 + 查询投影 + AI 视图裁剪**

核心思想是：

- **先确定当前问题所属的 `relation`（领域 / 检索域）**
- **再在该 `relation` 内，使用真实代码符号做检索**
- **底层存储尽量去冗余，不为 AI 展示而提前构造重嵌套结构**
- **通过反向查询、多关键词命中、group 过滤来缩小 `related_modules` 范围**
- **最终提供给 AI 的是投影结果，而不是原始 JSON 全量内容**

这个方案比“全量关系图”更轻，比“纯关键词搜索”更准，也比“把查询视图直接硬编码到存储结构里”更稳。

---

## 设计目标

### 1. 让存储层保持轻量和稳定

底层结构优先考虑去冗余、可维护、可增量更新，而不是直接为 AI 输出服务。

### 2. 让查询层承担结构转换职责

允许查询层根据当前问题，把底层结构投影成不同视图，而不是要求存储层一次性表达所有查询场景。

### 3. 避免 `related_modules` 膨胀成重嵌套结构

`related_modules` 只保留必要的关联引用，不重复嵌入完整模块对象。

### 4. 支持 relation 内的局部检索

同一个模块在不同 `relation` 中，可以拥有不同的关联模块集合。

### 5. 支持反向查询与多关键词查询

随着模块关系增加，需要通过“先命中模块、再反查关联”的方式控制返回规模。

### 6. 支持 group 过滤

在 `relation` 内进一步引入 group，对模块集合做二次切片，减少一次性提供给 AI 的无关模块。

### 7. 保持 AI 检索关键字可信

用于搜索的关键字必须来自真实代码证据，而不是 AI 自己概括出来的抽象词。

### 8. 支持失败停机与用户兜底

当 AI 无法判断领域或检索连续失败时，应主动停下并询问用户，而不是持续盲搜。

---

## 核心原则

## 一、存储层、查询层、AI 视图层分离

这是本方案最重要的原则。

### 1. 存储层

职责是：

- 保存稳定结构
- 避免冗余
- 方便更新
- 支持后续查询

存储层不追求“让 AI 直接读起来最顺”。

### 2. 查询层

职责是：

- 按 `relation` 限定范围
- 用真实关键字命中模块
- 做正向 / 反向关联查询
- 做 group 过滤
- 做排序、截断、聚合

查询层负责把底层结构转换成“当前问题需要的视图”。

### 3. AI 视图层

职责是：

- 进一步压缩查询结果
- 只保留当前问题所需的少量模块
- 补充必要的命中原因、证据来源和过滤说明

AI 最终看到的，应该是一个“**问题相关的结果视图**”，而不是原始存储结构全文。

---

## 二、`relation`

`relation` 在本方案中，仍然表示：

### **一个用于检索和记忆组织的领域 / 语义域 / 问题上下文域**

例如：

- `task-dispatch`
- `queue-routing`
- `worker-execution`
- `result-collection`
- `debug-timeout`

这些名字表达的是：

- 当前问题属于哪类业务链路或技术域
- 记忆应该优先从哪个范围读取
- 模块关系应该在哪个局部作用域内解释

因此，`relation` 的作用是：

- 限定搜索范围
- 为模块关系提供上下文
- 为 `solutions` 提供归档归属
- 为 group 提供上层边界
- 为后续增量补充提供落点

---

## 三、模块关系只在 relation 作用域内维护

模块关系不再尝试维护为整个项目的一张全局大图，而是：

### **在某个 `relation` 内部维护局部关系**

例如：

- `dispatcher` 在 `task-dispatch` 中的相关模块，可能是 `queue_router`、`scheduler`
- 同一个 `dispatcher` 在 `debug-timeout` 中的相关模块，可能又是 `retry_manager`、`timeout_guard`

这样可以避免：

- 一次查询返回所有领域下的模块
- 关系网无限增大
- 为了不同场景重复维护全局关系图

---

## 四、证据符号

证据符号是本方案的硬基础。

### **所有检索关键字都必须来自真实证据符号**

允许作为检索关键字的真实证据包括：

- 函数名，例如 `dispatch_task`
- 类名，例如 `QueueRouter`
- 方法名，例如 `QueueRouter.route`
- 文件相对路径，例如 `ab_celery/queues.py`
- 模块名
- 配置项名
- 队列名
- API 路径
- 错误消息原文
- 表名、字段名

不允许直接作为检索关键字的内容包括：

- AI 归纳出来但代码中不存在的抽象词
- 意义相近但不在代码中的同义改写
- “可能叫这个”的猜测词
- 仅来自语言概括、没有代码证据支撑的标签

换句话说：

- **`relation` 可以由 AI 基于证据进行归纳命名**
- **检索 `keyword` 不可以由 AI 凭空创造**

---

## 总体结构

建议 `project-memory` 的结构升级为以下几个核心文件：

```text
.project-memory/
├── index.json
├── relations.json
├── modules.json
├── relation-modules.json
├── relation-groups.json
├── solutions.md
├── code-map.md
├── architecture.md
├── decisions.md
├── work-log.md
└── glossary.md
```

其中：

- `relations.json`：记录所有 `relation` 的定义与候选切换策略
- `modules.json`：保存模块本体定义
- `relation-modules.json`：保存某个 `relation` 内的模块关联列表
- `relation-groups.json`：保存某个 `relation` 内的 group 划分
- `solutions.md`：记录带有 `relation` 和证据符号的问题解决经验
- `index.json`：统一维护条目级摘要索引
- 其他 Markdown 文件继续承担人类可读的正文存储职责

## 文件设计

## 一、`relations.json`

该文件用于定义当前系统已知的所有 `relation`。

它的作用不是存业务知识正文，而是作为：

### **领域选择器 / 检索域字典 / fallback 策略源**

### 推荐结构

```json
{
  "updated_at": "2026-05-20 16:00",
  "relations": [
    {
      "id": "task-dispatch",
      "name": "任务分发",
      "description": "任务编排、分发、路由选择相关内容",
      "fallback_relations": [
        "queue-routing",
        "worker-execution"
      ],
      "evidence_symbols": [
        "ab_celery/queues.py",
        "dispatch_task",
        "QueueRouter"
      ],
      "status": "active"
    }
  ]
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `id` | relation 唯一标识，建议稳定且简洁 |
| `name` | relation 展示名称 |
| `description` | 该领域负责的问题范围描述 |
| `fallback_relations` | 当前 relation 检索失败时可切换的候选 relation |
| `evidence_symbols` | 支撑该 relation 存在的真实代码符号 |
| `status` | 当前 relation 状态，如 `active`、`deprecated` |

### 设计原则

- `relation` 允许 AI 生成，但必须能追溯到真实证据
- `fallback_relations` 只表达“下一步可尝试的领域”，不表达模块边关系
- `description` 用于帮助 AI 判断当前问题更像哪个领域
- `evidence_symbols` 用于帮助 AI 在用户未明说领域时做高置信匹配

---

## 二、`modules.json`

该文件只保存模块本体定义。

它回答的是：

### **这个模块是谁**

而不是：

- 它在所有 relation 里和谁相关
- 它在当前问题里应该返回给 AI 什么结果

### 推荐结构

```json
{
  "updated_at": "2026-05-20 16:00",
  "modules": [
    {
      "id": "dispatcher",
      "name": "dispatcher",
      "aliases": ["DispatchService"],
      "evidence_symbols": [
        "dispatch_task",
        "DispatchService",
        "ab_celery/dispatcher.py"
      ]
    },
    {
      "id": "queue_router",
      "name": "queue_router",
      "aliases": ["QueueRouter"],
      "evidence_symbols": [
        "QueueRouter",
        "QueueRouter.route",
        "ab_celery/queues.py"
      ]
    }
  ]
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `id` | 模块唯一标识 |
| `name` | 模块主名称 |
| `aliases` | 模块别名 |
| `evidence_symbols` | 能定位该模块的真实代码符号 |

### 设计原则

- 模块定义只保存一份
- 模块本体信息不重复嵌入其他结构中
- 查询时优先通过 `module_id` 关联，而不是重复复制模块详情

---

## 三、`relation-modules.json`

该文件用于保存某个 `relation` 内的模块邻接关系。

### 关键调整

这里的 `related_modules` **不再使用完整对象结构**，而是直接使用列表。

也就是说：

- 只保留关联模块 ID
- 不在子项中重复存储完整模块定义
- 不在子项中塞入大量重复说明字段

### 推荐结构

```json
{
  "updated_at": "2026-05-20 16:00",
  "relations": [
    {
      "relation_id": "task-dispatch",
      "modules": [
        {
          "module_id": "dispatcher",
          "related_modules": [
            "queue_router",
            "scheduler",
            "retry_manager"
          ]
        },
        {
          "module_id": "queue_router",
          "related_modules": [
            "dispatcher",
            "queue_registry"
          ]
        }
      ]
    }
  ]
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `relation_id` | 当前局部索引所属的 relation |
| `module_id` | 当前模块 ID |
| `related_modules` | 与当前模块相关的模块 ID 列表 |

### 设计原则

- `related_modules` 直接用列表表达即可
- 子 `related_modules` 不再嵌完整结构，避免冗余
- 当前文件只表达“在这个 relation 里，它和谁相关”
- 关系解释、排序、证据拼装应尽量放到查询层，而不是全部压入存储层

---

## 四、`relation-groups.json`

该文件用于在某个 `relation` 内定义 group。

group 的作用是：

### **把同一个 relation 再拆成多个可过滤的局部模块集合**

这样可以在 `related_modules` 逐渐增多后，继续缩小结果范围。

### 推荐结构

```json
{
  "updated_at": "2026-05-20 16:00",
  "relations": [
    {
      "relation_id": "task-dispatch",
      "groups": [
        {
          "group_id": "dispatch-main-flow",
          "name": "主分发链路",
          "description": "任务分发主流程相关模块",
          "module_ids": [
            "dispatcher",
            "queue_router",
            "scheduler"
          ]
        },
        {
          "group_id": "retry-flow",
          "name": "重试链路",
          "description": "任务失败后的重试相关模块",
          "module_ids": [
            "dispatcher",
            "retry_manager",
            "dead_letter_handler"
          ]
        }
      ]
    }
  ]
}
```

### 设计原则

- group 和模块是多对多关系
- 一个模块可以属于多个 group
- 一个 group 可以包含多个模块
- group 不应强行嵌入 `related_modules` 子项内部
- group 独立维护后，查询时可通过交集过滤减少噪音

---

## 五、`solutions.md`

`solutions.md` 继续保留，但每条解决方案必须挂接到某个 `relation`，并明确其证据来源。

### 推荐格式

```markdown
### 队列路由错误导致任务进入错误队列
- **Date**: 2026-05-20 16:00
- **Relation**: task-dispatch
- **Groups**: dispatch-main-flow
- **Modules**: dispatcher, queue_router
- **Keywords**: dispatch_task, QueueRouter.route, ab_celery/queues.py
- **Evidence**: dispatch_task 调用了 QueueRouter.route，且路由表定义位于 ab_celery/queues.py

问题现象：
任务被分发到了错误队列。

原因：
路由匹配逻辑使用了错误的队列映射。

解决方案：
修正路由表配置，并验证 dispatch_task 的调用链。
```

### 设计原则

- 每条 solution 必须能回答“属于哪个 relation”
- 如有必要，可额外标注所属 group
- 每条 solution 必须能回答“用了哪些真实关键字定位问题”
- 每条 solution 必须能回答“证据来自哪里”

---

## 六、`index.json`

`index.json` 的职责从“纯条目元数据”升级为“统一条目索引”。

它不负责保存所有关系细节，只负责：

- 管理条目级元数据
- 给检索提供第一层概览
- 将条目与 `relation`、模块、group、关键字挂接起来

### 推荐结构

```json
{
  "updated_at": "2026-05-20 16:00",
  "entries": [
    {
      "category": "solutions",
      "title": "队列路由错误导致任务进入错误队列",
      "relation": "task-dispatch",
      "groups": ["dispatch-main-flow"],
      "modules": ["dispatcher", "queue_router"],
      "keywords": [
        "dispatch_task",
        "QueueRouter.route",
        "ab_celery/queues.py"
      ],
      "tags": ["queue", "dispatch"],
      "date": "2026-05-20 16:00",
      "pinned": false
    }
  ]
}
```

### 字段说明

| 字段 | 说明 |
|------|------|
| `category` | 条目所属分类 |
| `title` | 条目标题 |
| `relation` | 条目所属 relation |
| `groups` | 条目所属 group 列表 |
| `modules` | 条目关联模块 |
| `keywords` | 条目检索关键字，必须是真实符号 |
| `tags` | 补充标签 |
| `date` | 更新时间 |
| `pinned` | 是否钉住 |

### 设计原则

- `index.json` 用于快速查看条目概览，不保存复杂模块关系细节
- 每个条目都必须尽量带上 `relation`
- 当条目明确属于某个子场景时，也可以标注 `groups`
- `keywords` 必须是可检索的真实符号，而不是 AI 归纳摘要词

---

## 查询模型设计

本方案的关键不只是结构，而是：

### **检索流程必须改成 relation-first，并支持多关键词 + 反向查询 + group 过滤**

---

## 阶段一：先判断当前问题属于哪个 `relation`

AI 在开始检索前，应优先判断：

### **当前问题最可能属于哪个 relation**

判断依据包括：

- 用户是否明确给出了领域名称
- 用户问题中是否包含某些真实代码符号
- 这些符号是否能和 `relations.json` 中的 `evidence_symbols` 匹配
- 某个 relation 的 `description` 是否与当前问题高度吻合
- 某个模块是否长期稳定地出现在某个 relation 下

如果某个 relation 的置信度足够高，则进入下一步。

如果没有足够高的候选 relation，则进入停机询问流程。

---

## 阶段二：使用多个真实关键词命中模块

确定 relation 后，不应立即把整个 `relation` 下的所有 `related_modules` 一次性都给到 AI。

正确做法是先使用：

### **多个真实关键词去命中模块**

例如：

- `dispatch_task`
- `QueueRouter.route`
- `ab_celery/queues.py`

这些关键词必须来自：

- 用户显式提到的真实符号
- `relations.json` 中已有的 `evidence_symbols`
- `index.json` 中条目的 `keywords`
- 当前任务真实读到的代码符号

### 匹配优先级建议

1. `modules.json` 中的 `evidence_symbols`
2. `modules.json` 中的 `aliases`
3. `modules.json` 中的 `name`
4. `index.json` 中的 `keywords`

### 多关键词策略建议

- 优先使用 AND 思路提升精度
- 如命中过少，再退化到 OR 扩大召回
- 匹配结果最终统一映射为 `module_id`

---

## 阶段三：通过正向与反向查询确定候选 `related_modules`

当关键词已经命中了某些 `module_id` 后，再进入关联查询。

### 1. 正向查询

查询：

- 当前命中模块的 `related_modules`

例如：

- 命中 `dispatcher`
- 查询 `dispatcher.related_modules`

### 2. 反向查询

查询：

- 哪些模块的 `related_modules` 中包含当前模块

例如：

- 查询谁把 `dispatcher` 当作相关模块

### 设计意义

这样做的好处是：

- 不需要一次性展开整个 relation 下的所有模块关系
- 可以根据真实关键词先定位局部模块，再反查周边模块
- 适合随着时间推移 `related_modules` 持续增长的场景

---

## 阶段四：如果有 group，则再做一次过滤

当用户显式指定了 group，或者系统可以高置信推断出某个 group 时，应对候选 `related_modules` 做一次 group 过滤。

### 过滤方式

- 先得到候选 `related_modules`
- 再与当前 group 的 `module_ids` 做交集

例如：

候选模块是：

```json
["queue_router", "scheduler", "retry_manager", "dead_letter_handler"]
```

如果 group 为 `dispatch-main-flow`，则过滤后变为：

```json
["queue_router", "scheduler"]
```

如果 group 为 `retry-flow`，则过滤后变为：

```json
["retry_manager", "dead_letter_handler"]
```

### 设计意义

- group 是 relation 内的第二层切片
- group 能显著减少一次性提供给 AI 的无关模块
- group 与模块多对多关系更适合独立维护，而不是硬编码到 `related_modules` 子项里

---

## 阶段五：把查询结果投影为 AI 视图

这一步非常关键。

### **AI 最终看到的不是原始存储结构，而是查询投影结果**

推荐的 AI 视图可以类似：

```text
relation: task-dispatch
matched_modules:
- dispatcher
  evidence: dispatch_task, ab_celery/dispatcher.py

related_modules:
- queue_router
  matched_by: forward_related + group(dispatch-main-flow)
- scheduler
  matched_by: forward_related + group(dispatch-main-flow)
```

也就是说，AI 视图层只保留：

- 当前 relation
- 命中的模块
- 最终筛选后的相关模块
- 命中原因
- 必要证据

而不会把整个 `modules.json`、`relation-modules.json`、`relation-groups.json` 全量原样传给 AI。

---

## 阶段六：第一次没命中时，尝试 fallback relation

当当前 relation 中没有得到有效结果时，不应立即扩展到全局搜索，而应优先参考：

- `relations.json` 中的 `fallback_relations`

例如：

- 当前 relation 是 `task-dispatch`
- fallback 依次为 `queue-routing`、`worker-execution`

则应按顺序切换 relation，再重复进行：

- 多关键词命中模块
- 正向 / 反向关联查询
- group 过滤
- AI 视图投影

---

## 阶段七：三次失败停机并询问用户

当发生以下任一情况时，视为一次失败：

- 当前 relation 下完全没有命中模块
- 命中的模块与用户目标代码明显不相关
- 正向 / 反向查询后仍无法定位有效 `related_modules`
- 切换 fallback relation 后结果依然不符合预期

### 硬规则

### **连续 3 次 relation-scoped 检索失败后，必须主动停下来询问用户**

应直接询问：

- **这个问题属于哪个领域（relation）？**
- **如果不知道 relation，请提供真实文件、类名、函数名、错误信息或代码片段**

---

## 写入流程设计

## 一、写入普通条目时的规则

无论写入 `code-map`、`architecture` 还是 `solutions`，都建议按照如下步骤：

1. 判断当前条目是否能归属到某个 relation
2. 提取真实模块名与真实关键字
3. 更新 `index.json`
4. 如有必要，更新 `modules.json`
5. 如有必要，更新 `relation-modules.json`
6. 如有必要，更新 `relation-groups.json`
7. 如是问题解决经验，更新 `solutions.md`
8. 如当前 relation 尚不存在，则先生成或补充 `relations.json`

---

## 二、写入 `relation` 的规则

只有在下面几种情况下，才创建新的 relation：

- 当前问题明确不属于已有 relation
- 已有 relation 连续检索失败，且用户补充了足够真实上下文
- 当前代码改动反复集中在一类稳定业务域上，值得长期沉淀

### 不应创建新 relation 的情况

- 只是临时小问题，没有领域稳定性
- 没有足够的真实证据符号
- 只是 AI 自己感觉“好像应该分一个类”

---

## 三、写入模块的规则

只有在确认模块在系统中尚未存在，或已有定义需要补充时，才更新 `modules.json`。

### 适合写入模块定义的情况

- 识别出了新的稳定模块实体
- 获取到了新的别名
- 获取到了新的真实证据符号

### 不应写入模块定义的情况

- 只是模糊提到一个可能的名字
- 无法确定是否为稳定模块
- 只是临时日志片段中的不可靠字符串

---

## 四、写入模块关系的规则

只有在当前任务能明确支撑模块关系时，才补充 `relation-modules.json`，例如：

- 读到了明确调用链
- 看到了配置引用关系
- 看到了条目中稳定重复出现的模块共现
- 用户明确确认了某模块与另一模块的关系

### 不应补充模块关系的情况

- 只是猜测“它们应该相关”
- 只是同一目录出现过
- 只是名字相似
- 没有具体证据支撑

### 重要约束

- `related_modules` 只追加模块 ID
- 不在 `related_modules` 子项里重复塞模块详情
- 不为了方便 AI 直接展示而提前构造重嵌套结构

---

## 五、写入 group 的规则

只有在当前任务能明确识别出 relation 内部稳定的子场景时，才补充 `relation-groups.json`。

### 适合写入 group 的情况

- 主流程与重试流程明显分离
- 初始化流程与运行时流程明显分离
- 读写链路、同步链路、排障链路具有稳定边界

### 不应写入 group 的情况

- 只是一次性临时查询集合
- 只是某次问题中临时拼凑的结果
- 还无法确认该集合会被长期复用

---

## 日常使用中的增量补充策略

这套方案不应只在“专门维护记忆”时更新，还应在日常任务中自然演进。

### 推荐策略

当平时对某个领域进行修改时，例如：

- 修改 `task-dispatch` 相关代码
- 修复 `worker-timeout-debug` 相关问题
- 处理 `queue-routing` 逻辑

如果当前任务中读到了新的明确证据，则可以顺手补充：

- relation 的描述
- relation 的证据符号
- relation 的 fallback relation
- 模块定义
- relation 下的模块邻接关系
- relation 下的 group
- 该 relation 的 solution 条目

### 原则

### **只补充已经被当前任务证实的内容，不补充猜测内容**

---

## 与上一版方案的差异

上一版方案虽然已经改成了 `relation-first`，但还有两个问题没有完全收紧：

### 1. 没有显式区分“存储结构”和“AI 看到的结果”

这会导致后续设计不断把查询层职责塞回存储层。

### 2. `related_modules` 仍然过于接近“完整子结构”

虽然它表达力强，但后续会带来明显冗余。

### 3. 缺少反向查询与 group 过滤的明确模型

如果未来 `related_modules` 不断增长，没有这两层机制，最后仍然会一次性返回太多模块给 AI。

因此，本次调整后的重点变成：

- 存储层归一化
- 查询层负责转换
- `related_modules` 只保留列表
- 支持反向查询
- 支持 group 多对多过滤
- AI 最终只看查询投影结果

---

## 推荐的执行规则总结

### 规则一：relation 优先

所有检索优先先判断 relation，再进入 relation 内检索。

### 规则二：关键字必须是真实符号

所有用于搜索的关键字必须来自代码、路径、类名、方法名、错误原文等真实证据。

### 规则三：存储结构不等于 AI 视图

底层结构只负责稳定存储，AI 最终应看到裁剪后的查询结果。

### 规则四：`related_modules` 只保留列表

子关联不再嵌完整结构，模块详情独立存储。

### 规则五：优先通过多关键词命中模块，再做正反向查询

不要直接把整个 relation 下的所有关联模块全部展开。

### 规则六：group 作为 relation 内第二层过滤

group 与模块是多对多关系，应独立维护，并在查询时做交集过滤。

### 规则七：三次失败必须停机询问

连续 3 次 relation-scoped 检索失败后，必须主动向用户确认领域，或索取更多真实上下文。

### 规则八：只写入被证实的关系

模块关系、领域归属、group、解决方案都应由当前任务中的真实证据支撑。

## 推荐演进顺序

### 第一阶段：建立归一化存储结构

先新增并维护：

- `relations.json`
- `modules.json`
- `relation-modules.json`
- `relation-groups.json`
- `index.json` 中的 `relation`、`groups` 和 `keywords`

### 第二阶段：改造检索流程

让搜索过程强制变成：

- 判断 relation
- 多关键词命中模块
- 正向 / 反向关联查询
- group 过滤
- AI 视图投影
- fallback relation
- 三次失败停机询问

### 第三阶段：改造写入流程

让新增条目时自动尝试：

- 补充 relation
- 提取真实关键字
- 更新模块定义
- 更新 relation 模块邻接关系
- 更新 group

### 第四阶段：支持 relation 自动生长

当用户提供足够上下文时，允许 AI 创建新的 relation，并自动生成初始：

- relation 定义
- 模块定义
- 模块邻接关系
- group 切片

## 结论

这套调整后的方案，核心不是构建“最完整的关系图”，也不是构建“AI 直接可读的超大嵌套 JSON”，而是构建：

### **一个以 `relation` 为第一检索入口、以真实代码符号为检索依据、以归一化存储为底座、以查询投影为中间层、以 AI 视图裁剪为最终输出的项目记忆系统**

它有几个关键优势：

- **更轻**：底层结构更去冗余
- **更稳**：模块定义和模块关系分开维护
- **更准**：先命中模块，再查关联，再做过滤
- **更可控**：group 能进一步缩小结果范围
- **更适合 AI**：AI 看到的是裁剪后的结果，而不是原始存储全文
- **更适合演进**：后续可以持续增加 relation、模块、group，而不会快速失控

因此，建议 `project-memory` 的后续设计以这套方案为主：

- 强制区分存储层、查询层、AI 视图层
- 保留 `relations.json` 作为领域词典
- 新增 `modules.json` 作为模块本体定义
- 在 `relation-modules.json` 中只维护模块 ID 列表形式的局部邻接关系
- 在 `relation-groups.json` 中维护 relation 内的 group 切片
- 查询时通过多关键词命中、正向 / 反向查询、group 过滤缩小范围
- 最终只把投影后的少量高相关结果提供给 AI

这会比上一版更贴近你刚刚提出的三个关键约束，也更适合后续真正落地。
