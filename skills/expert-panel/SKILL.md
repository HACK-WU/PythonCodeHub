# Expert Panel（专家团）

通过多agent角色协作，对方案进行结构化评审，克服单AI决策的确认偏误和视角盲区。

## 核心设计理念

- **角色分离**：主agent在"执行者"和"法官"之间切换，避免自我确认
- **独立质疑**：两个反叛者相互隔离，确保质疑的独立性和多样性
- **群体评分**：评委可讨论但独立打分，提供多维度量化参考
- **过程透明**：最终报告包含所有原始材料，用户可见完整辩论过程
- **文件持久化**：所有产出物以文件形式落盘，子agent按需读取，避免prompt膨胀
- **Skill可组合**：专家团是协作框架，不排斥外部skill。主agent在执行阶段应主动发现并使用相关skill；用户明确指定skill时优先遵从

## 文件持久化机制

### 目录结构

```
.codebuddy/expert-panel/{task-name}/
├── scheme.md              # 方案全文（简单方案单文件）
├── scheme/                # 复杂方案拆分目录（与scheme.md二选一）
│   ├── overview.md        # 方案总览（入口文件）
│   ├── architecture.md    # 架构设计
│   ├── api-design.md      # 接口设计
│   └── ...                # 按需拆分
├── code-context.md        # 代码上下文索引
├── design-decisions.md    # 设计决策记录
├── rebel-output/          # 反叛者产出
│   ├── rebel-a.md
│   └── rebel-b.md
├── judge-output/          # 评委产出
│   ├── judge-1.md
│   ├── judge-2.md
│   └── judge-3.md
└── final-report.md        # 最终评审报告
```

### 规则

- **简单方案** → 单文件 `scheme.md`
- **复杂方案** → 拆分到 `scheme/` 目录，`overview.md` 为入口
- 主agent自行判断复杂度选择方式
- **文件不自动删除**，保留为评审过程的完整记录，供用户随时查阅

### task-name 命名规则

- **英文小写 + 短横线**，从任务描述中提取 2-4 个关键词
- 示例：`jwt-auth-review`、`user-api-refactor`、`payment-flow-design`
- 总长度 ≤ 64 字符
- 如遇同名目录，追加 `-v2` 后缀

### 信息分层传递原则

| 层级 | 内容 | 传递方式 |
|------|------|----------|
| 第一层 | 方案全文、设计决策 | 文件落盘 → 子agent按需读取 |
| 第二层 | 代码上下文索引（路径+摘要） | 文件落盘 → 子agent按需读取，需要时再read_file具体代码 |
| 第三层 | 具体代码实现 | 子agent根据索引自行读取，不在文件中展开 |

## 评审专家团完整流程

```
用户提出任务
    │
    ▼
【阶段0：Skill发现】（可选）
    │  用户是否指定了skill？
    │  ├── 是 → 直接加载指定skill
    │  └── 否 → 主agent扫描可用skill列表，判断是否有相关的skill
    │       ├── 有 → 加载相关skill，按其规范执行
    │       └── 无 → 正常进入阶段1
    │
    ▼
【阶段1：执行】主agent（执行者模式）
    │  自由创建子agent完成任务（如已加载skill则在其规范下执行）
    │  产出文件：
    │  ├── scheme.md（或scheme/目录）
    │  ├── code-context.md
    │  └── design-decisions.md
    │
    ▼
【阶段2：质疑】主agent角色转换 → 法官模式
    │  创建团队，并行启动两个反叛者
    │  prompt只含文件路径列表
    │
    ├─→ [反叛者A] 读取文件 → 独立质疑 → 写入 rebel-a.md
    └─→ [反叛者B] 读取文件 → 独立质疑 → 写入 rebel-b.md
        （两者相互隔离，看不到对方的输出）
    │
    ▼
【阶段3：评分】主agent启动评委团
    │  prompt含方案+索引+决策+反叛者产出的文件路径列表
    │
    ├─→ [评委1] 读取文件 → 打分 → 写入 judge-1.md
    ├─→ [评委2] 读取文件 → 打分 → 写入 judge-2.md ─┐
    └─→ [评委3] 读取文件 → 打分 → 写入 judge-3.md ─┤ 可讨论
    │
    ▼
【阶段4：研判】主agent（法官模式）
    │  读取所有产出文件 → 判断终止条件
    │
    ▼
【阶段5：输出】完整评审报告交付用户
    │  文件保留在 .codebuddy/expert-panel/{task-name}/ 供查阅
```

### 阶段0：Skill发现与选择（可选，阶段1前执行）

- 主agent检查用户是否明确指定了要使用的skill（如"用design-doc-generator写设计文档"）
  - **用户指定** → 直接调用 `use_skill` 加载指定skill，按其规范执行
  - **用户未指定** → 主agent扫描当前可用skill列表（`<available_skills>` 中列出的skill），根据任务描述判断是否存在相关skill
    - **发现相关skill** → 调用 `use_skill` 加载，在其规范指导下产出方案
    - **无相关skill** → 正常进入阶段1
- **注意**：此步骤仅主agent执行，子agent（反叛者/评委）不涉及skill调用

### 阶段1：执行——主agent自由产出方案V0

- 主agent以**执行者身份**完成任务，可自由创建任意数量子agent辅助
- 不限制角色、不限制方式，目标是产出高质量方案
- **产出文件**（必须写入磁盘）：
  - `scheme.md`（或 `scheme/` 目录）：方案全文
  - `code-context.md`：代码上下文索引
  - `design-decisions.md`：设计决策记录
- 主agent在执行过程中每查阅一个代码文件，顺势将路径和摘要记录到 `code-context.md`，避免事后回忆遗漏
- 完成后，主agent身份切换为**法官**，进入阶段2

**scheme.md 建议结构**（非强制，主agent根据任务调整）：

```markdown
# {任务名称} 方案

## 背景与目标
## 方案概述
## 详细设计
## 技术选型与决策
## 风险与应对
## 实施计划
```

**code-context.md 格式**：

```markdown
# 代码上下文索引

| 文件/模块 | 关键信息摘要 | 评审相关度 |
|-----------|-------------|-----------|
| src/api/user.py | 用户认证接口，JWT签发逻辑 | 高 |
| src/middleware/auth.py | 鉴权中间件，token校验 | 高 |
| src/models/user.py | User模型定义 | 中 |
| src/config.py | 配置项，含token过期时间 | 低 |
```

**design-decisions.md 格式**：

```markdown
# 设计决策记录

- 为什么选JWT而非Session：[原因]
- 为什么token过期时间设30分钟：[原因]
- 为什么不用OAuth2：[原因]
```

### 阶段2：质疑——双反叛者独立审查

**角色定位**：反叛者是"严谨的挑战者"，不是"杠精"。目标是找出方案的不合理之处，而非无差别攻击。

**创建方式**：使用 `team_create` 创建评审团队，然后用 `Task` 并行启动两个反叛者子agent。

**信息传递（主agent → 反叛者prompt）**：

```
你作为反叛者（Rebel Reviewer），你的职责是：
1. 竭尽全力找出方案中的不合理之处、漏洞、风险
2. 每条反对意见必须附带推理链条，说明"为什么不好"
3. 每条反对意见必须给出对应的替代方案
4. 也必须列出你对原方案的认可点

请先读取以下文件，然后进行审查：

| 文件 | 内容说明 | 优先级 |
|------|----------|--------|
| .codebuddy/expert-panel/{task-name}/scheme.md | 方案全文 | 必读 |
| .codebuddy/expert-panel/{task-name}/code-context.md | 代码上下文索引 | 必读 |
| .codebuddy/expert-panel/{task-name}/design-decisions.md | 设计决策记录 | 必读 |

如方案拆分为目录，请先读取 scheme/overview.md 了解全貌，再按需深入各子文件。
如需查看具体代码，根据 code-context.md 中的路径自行读取。

请将审查结果写入：.codebuddy/expert-panel/{task-name}/rebel-output/rebel-a.md
```

**反叛者产出格式（写入文件）**：

```markdown
## 反对意见

### 意见1：[标题]
- 反对理由：（推理链条，为什么这是个问题，什么场景下会暴露）
- 严重程度：🔴 致命 / 🟡 重要 / 🟢 建议
- 替代方案：（针对此条的具体替代做法）

### 意见2：[标题]
- 反对理由：...
- 严重程度：...
- 替代方案：...

## 认可点
- （方案中做得好的方面，至少1条）

## 整体替代方案（可选）
（仅当你认为整个方案方向有问题时，才提出完整替代方案）
```

**约束**：
- 反叛者A和反叛者B相互完全隔离，不知道对方的存在和输出
- 两条反对意见中如果出现了相同的点，说明问题严重，在最终报告中需高亮为"共识性反对"

### 阶段3：评分——三评委独立打分 + 讨论

**角色定位**：评委是"公正的评估者"，综合考虑原始方案和各反对意见后给出量化评价。

**创建方式**：用 `Task` 并行启动3个评委子agent，同属一个team。

**信息传递（主agent → 各评委prompt）**：

```
你作为评委（Judge），请对方案和反对意见进行评审。
你可以与其他评委讨论，但最终打分由你独立决定。

请先读取以下文件：

| 文件 | 内容说明 | 优先级 |
|------|----------|--------|
| .codebuddy/expert-panel/{task-name}/scheme.md | 原始方案全文 | 必读 |
| .codebuddy/expert-panel/{task-name}/code-context.md | 代码上下文索引 | 必读 |
| .codebuddy/expert-panel/{task-name}/design-decisions.md | 设计决策记录 | 必读 |
| .codebuddy/expert-panel/{task-name}/rebel-output/rebel-a.md | 反叛者A完整审查意见 | 必读 |
| .codebuddy/expert-panel/{task-name}/rebel-output/rebel-b.md | 反叛者B完整审查意见 | 必读 |

如需查看具体代码，根据 code-context.md 中的路径自行读取。

评分维度（1-5分）：
- 可行性：方案在技术上是否可实现？（1=不可行，5=完全可行）
- 风险覆盖：方案是否覆盖了主要风险点？（1=风险敞口大，5=风险可控）
- 完整性：方案是否覆盖了所有场景和边界情况？（1=遗漏严重，5=覆盖全面）
- 采纳度：方案对反叛者合理意见的采纳程度？（1=完全不采纳，5=充分采纳）

请将评分和理由写入：.codebuddy/expert-panel/{task-name}/judge-output/judge-1.md
```

**评委讨论机制**：
- 评委之间可通过 `send_message` 相互讨论
- 不指定侧重领域，每个评委从所有评分维度全面评估
- 通过讨论聚焦争议点，最终独立给出分数
- 每个评委独立给出最终分数，写入各自文件

### 阶段4：研判——主agent判断终止条件

主agent读取所有 `judge-output/` 和 `rebel-output/` 文件，检查是否满足**任一**终止条件：

| 条件 | 说明 |
|------|------|
| 评分收敛 | 各维度评分中，任意维度的最高分与最低分差距 ≤ 1 |
| 覆盖完整 | 所有方案（V0 + 替代方案A/B）都已获得评分 |
| 讨论轮次 | 评委讨论已达 2-3 轮 |
| 意见枯竭 | 新增讨论不再产生实质性新意见 |

满足任一条件 → 评委讨论终止，进入阶段5。否则 → 回到阶段3继续讨论。

- 完成后，调用 `team_delete()` 清理团队

### 阶段5：输出——完整评审报告

主agent读取所有产出文件，汇总后按以下结构向用户交付最终报告：

- 评审报告直接以对话形式输出给用户
- 报告同时写入 `.codebuddy/expert-panel/{task-name}/final-report.md`
- 用户可随时查看中间产物文件

```markdown
# 评审报告：{任务名称}

## 一、原始方案
（读取 scheme.md 或 scheme/overview.md 展示）

## 二、反叛者审查意见

### 反叛者A
（读取 rebel-a.md 展示：反对意见 + 逐条替代方案 + 认可点）

### 反叛者B
（读取 rebel-b.md 展示：反对意见 + 逐条替代方案 + 认可点）

### 共识性反对
（A和B独立发现的相同问题，需高亮标注，说明问题严重性）

## 三、评委评分与理由

| 评委 | 可行性 | 风险覆盖 | 完整性 | 采纳度 | 综合意见 |
|------|--------|----------|--------|--------|----------|
| 评委1 | X/5 | X/5 | X/5 | X/5 | 简要理由 |
| 评委2 | X/5 | X/5 | X/5 | X/5 | 简要理由 |
| 评委3 | X/5 | X/5 | X/5 | X/5 | 简要理由 |

**评委讨论要点**：（如有讨论，摘录关键分歧和共识）

## 四、主agent最终判断

- **采纳**：哪些反叛者意见被采纳，为什么
- **拒绝**：哪些反叛者意见未被采纳，为什么
- **存疑**：哪些问题需要用户最终决策
- **改进建议**：综合各方意见后的优化方向

## 五、评审过程文件
所有中间产物保存在：`.codebuddy/expert-panel/{task-name}/`
用户可直接查看各文件的完整内容。
```

## 执行专家团（Task Force）

执行专家团模式用于"做任务"而非"评审任务"，将在后续讨论中详细设计。

**占位说明**：
- 定位：主agent将复杂任务拆分为子任务，分发各领域专家并行执行
- 与评审专家团的区别：评审是"评估已有产出"，执行是"创造新产出"
- 共享文件持久化机制：执行产出同样落盘到 `.codebuddy/expert-panel/{task-name}/`
- 可复用评审专家团的异常处理、循环控制等机制
- 当前skill暂时只支持评审专家团模式

---

## 异常处理与边界情况

- **子agent失败**：单个反叛者/评委失败时，主agent可选择重试或跳过该角色继续流程
- **产出为空**：子agent未产出内容时，主agent记录"该角色未响应"，不影响其他角色继续
- **超时控制**：建议单个子agent执行时间不超过5分钟，超时由主agent主动终止
- **用户中断**：用户随时可要求停止评审，主agent清理团队并输出当前阶段的中间结果

---

## 循环控制机制

- 评委讨论最多进行3轮（初始1轮 + 追加2轮）
- 每轮讨论后主agent检查终止条件
- 需要继续时，主agent通过 `send_message` 向各评委发送"请继续讨论"指令，并附上需要聚焦的问题
- 各评委更新自己的 `judge-{n}.md` 文件
- 如需继续讨论，回到阶段3

---

## 角色设计原则

- **反叛者**：不指定侧重领域，两个反叛者全面审查。独立发现相同问题时标记为"共识性反对"
- **评委**：不指定侧重，每个评委从所有评分维度评估。通过讨论聚焦争议点
- **理由**：独立重复是验证质量的手段，指定侧重可能创造新的盲区

## 技术实现指引

### 所用工具

| 工具 | 用途 |
|------|------|
| `team_create` | 创建评审团队（阶段2开始时） |
| `Task`（team模式） | 启动反叛者和评委子agent |
| `send_message` | 评委间讨论、子agent回报结果 |
| `team_delete` | 评审完成后清理团队 |
| `write_to_file` | 主agent/子agent将产出写入文件 |
| `read_file` | 子agent按需读取方案、代码等文件 |

### 关键规则

1. **反叛者必须并行启动**：两个反叛者之间绝对不能有信息泄露
2. **评委必须并行启动**：但允许启动后通过 send_message 相互讨论
3. **文件持久化**：所有产出物必须写入文件，prompt中只传路径，不传内容
4. **信息分层**：方案和决策完整落盘；代码只存索引（路径+摘要），子agent自行按需读取
5. **方案拆分**：主agent根据复杂度自行决定单文件 or 目录拆分，子agent先读 overview
6. **反叛者替代方案逐条绑定**：每条反对意见必须附带对应替代方案，而非笼统一个
7. **完成后清理**：调用 `team_delete` 清理团队，但文件保留供用户查阅
8. **循环控制**：评委讨论最多3轮，每轮后检查终止条件
9. **异常处理**：单个子agent失败不阻塞整体流程，可重试或跳过
10. **输出规范**：评审报告同时输出给用户和写入 `final-report.md`

### 注意事项

- 3评委配置是默认值，用户可覆盖
- 评审流程中的阶段跳过或合并由主agent根据任务复杂度自行决策
- task-name 从任务描述提取，英文小写+短横线，如 `jwt-auth-review`
- task-force模式暂不启用

## 使用示例

### 触发方式

用户可以通过以下方式触发评审专家团：

- "帮我评审一下这个设计方案"
- "review 一下这个代码"
- "找专家看看这个架构"
- "全面评估一下这个方案"
- "看看有什么问题"

### 典型流程

1. 用户：帮我评审一下 JWT 认证方案
2. 主agent：扫描可用skill → 发现无相关skill → 进入阶段1
3. 主agent：产出方案文件（scheme.md、code-context.md、design-decisions.md）
4. 主agent：创建团队，启动两个反叛者
5. 反叛者A/B：独立审查，产出 rebel-a.md、rebil-b.md
6. 主agent：启动三个评委
7. 评委：讨论评分，产出 judge-1.md、judge-2.md、judge-3.md
8. 主agent：研判终止条件，输出最终报告
9. 用户：收到完整评审报告，可查看所有中间文件

### 查看中间产物

所有评审过程文件保存在：`.codebuddy/expert-panel/{task-name}/`

用户可直接打开查看：
- 原始方案：`scheme.md`
- 反叛者意见：`rebel-output/rebel-a.md`、`rebel-output/rebel-b.md`
- 评委评分：`judge-output/judge-1.md`、`judge-output/judge-2.md`、`judge-output/judge-3.md`
- 最终报告：`final-report.md`
