## GitNexus 使用手册（技能精简版）

### 定位

GitNexus 是面向代码库的知识图谱工具，适合做**结构化理解**和**影响分析**。
如果只是找精确字符串、字段名、枚举值、字面量引用，优先用普通文本搜索；如果要看**谁依赖谁、调用链、影响范围、执行流**，优先用 GitNexus。

### 适用场景

- 理解陌生代码库的结构和核心流程
- 查询某个类、函数、资源对象的上下游关系
- 修改前评估影响范围
- 做跨模块重构或重命名
- 提交前检查改动风险

### 不适合的场景

- 精确查找字符串、配置键、枚举值、常量值
- 确认某段实现细节是否真的存在
- 查找属性访问型引用、简单文本拼接引用

这类情况通常直接做文本搜索更快、更准。

### 最小使用流程

#### 1. 先保证索引可用

常用命令：

```bash
gitnexus analyze
gitnexus status
gitnexus analyze --force
```

含义：

- `gitnexus analyze`：建立或更新索引
- `gitnexus status`：检查索引是否过期
- `gitnexus analyze --force`：结果异常时强制全量重建

#### 2. 再根据任务类型选工具

- **理解某个符号**：`context`
- **评估改动影响**：`impact`
- **自定义关系追踪**：`cypher`
- **提交前看改动风险**：`detect_changes`
- **跨文件重命名**：`rename`
- **按主题检索流程**：`query`

### 工具选择速查

#### `context`

用途：查看某个符号的完整上下文。
重点关注：

- 符号定义位置
- 谁引用它（incoming）
- 它调用了谁（outgoing）
- 它参与了哪些执行流（processes）

适合：

- 已知具体类名/函数名
- 想快速看“上下游依赖”
- 改代码前先建立局部上下文

注意：

- 常见方法名可能有重名，需要结合文件路径或其他上下文消歧
- 已知具体符号时，通常优先于模糊检索

#### `impact`

用途：分析修改某个符号后的影响半径。
常用维度：

- `upstream`：谁依赖我
- `downstream`：我依赖谁
- `maxDepth`：追踪深度
- `relationTypes`：关系类型过滤

适合：

- 修改类、服务、资源对象前先评估风险
- 判断一个改动会波及哪些模块
- 提交前补做爆炸半径分析

注意：

- 对类级符号通常效果很好
- 对枚举常量、属性访问、部分动态引用覆盖不一定完整
- 这类盲区要配合文本搜索补齐

#### `cypher`

用途：做原生图查询，适合高级追踪。
适合：

- `context` 或 `impact` 结果还不够
- 需要查自定义调用链、热点符号、入口点
- 需要按特定关系拼出更复杂的问题

建议：

- 作为补充工具使用
- 当你已经知道要追什么关系时最有价值

#### `detect_changes`

用途：把当前代码改动映射到受影响的符号和执行流。
适合：

- 提交前风险检查
- 评估当前 diff 是否波及关键流程

最佳实践：

1. 代码改完后先看风险等级
2. 如果风险中高，再继续做 `impact` 分析
3. 确认安全后再提交

#### `rename`

用途：基于图谱 + 文本搜索做跨文件重命名。
推荐流程：

1. 先 `context` 看上下游
2. 再 `rename(dry_run=true)` 预览
3. 人工复核低置信度文本替换
4. 再正式执行

注意：

- 图谱命中的编辑可信度高
- 文本搜索补到的编辑要人工复核

#### `query`

用途：按主题做混合检索，并按执行流聚合结果。
适合：

- 还不知道具体符号名
- 想先按“认证流程”“告警处理”“数据查询”这类主题入手

限制：

- 当关键词过宽泛时，可能混入无关内容
- 已知具体类名/函数名时，不如直接查 `context`

### 推荐决策规则

#### 规则 1：精确引用优先文本搜索

以下内容优先文本搜索，不要首先依赖 GitNexus：

- 字段名
- 枚举值
- 常量
- 配置键
- 简单字符串引用
- 属性访问型引用

#### 规则 2：结构关系优先 GitNexus

以下问题优先 GitNexus：

- 谁调用了它
- 它依赖了谁
- 它在哪些流程里出现
- 改这个会影响什么
- 这个模块和哪个社区/流程相关

#### 规则 3：复杂任务混合使用

推荐组合：

- **先文本搜索定位，再用 `context` 建立结构上下文**
- **先 `impact` 看风险，再用文本搜索补漏**
- **先 `query` 找主题入口，再用 `context` 深挖关键符号**
- **先 `detect_changes` 看改动范围，再决定是否做 `impact`**

### 面向 AI 的推荐工作流

#### 工作流 1：理解陌生功能

1. 先确认索引可用
2. 用 `query` 搜功能主题
3. 找到关键符号后用 `context`
4. 如果要梳理完整链路，再补 `cypher`
5. 对关键引用再做文本搜索确认实现细节

#### 工作流 2：改代码前做影响评估

1. 先锁定要修改的类或函数
2. 用 `context` 看上下游
3. 用 `impact` 看影响半径
4. 对动态引用、属性引用、常量引用再做文本搜索补漏
5. 再开始修改

#### 工作流 3：提交前风险检查

1. 运行 `detect_changes`
2. 看 `risk_level`、`changed_symbols`、`affected_processes`
3. 如果风险较高，再对关键符号执行 `impact`
4. 必要时补做文本搜索复核

#### 工作流 4：重命名或重构

1. 用 `context` 先看符号周边依赖
2. 用 `rename` 先 dry run
3. 重点检查文本搜索补到的变更
4. 再执行正式重命名
5. 修改后重新检查索引状态

### 常用命令速查

```bash
gitnexus analyze
gitnexus analyze --force
gitnexus analyze --repair-fts
gitnexus status
gitnexus list
gitnexus clean
```

补充：

- `--repair-fts`：仅修复全文检索索引
- `list`：查看已索引仓库
- `clean`：删除索引

### 常见问题处理

#### 查询结果不完整

优先顺序：

1. `gitnexus status`
2. `gitnexus analyze`
3. `gitnexus analyze --force`
4. 如果只是搜索异常，再尝试 `gitnexus analyze --repair-fts`

#### 首次启动慢或超时

可先做一次预热：

```bash
npx gitnexus@latest --help
```

如果长期使用，优先全局安装并配置。

#### 修改后结果不对

先怀疑索引过期，再考虑工具本身限制。
对动态调用、枚举常量、属性访问等问题，不要只相信图谱结果，必须补做文本确认。

### 一句话结论

**GitNexus 负责“结构理解与影响分析”，文本搜索负责“精确定位与实现确认”，高质量代码探索通常要两者结合。**
