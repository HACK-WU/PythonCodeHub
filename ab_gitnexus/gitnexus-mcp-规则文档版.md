## GitNexus MCP 强制规则

### 总体规则

- 当任务目标是**理解代码结构、分析依赖关系、评估改动影响**时，**必须优先考虑 GitNexus 能力**，而不是只做普通文本搜索。
- 当任务目标是**定位字符串、字段名、枚举值、常量、配置键、属性访问或具体实现代码**时，**必须优先使用 grep / 文本搜索**，不得将这类问题错误地交给 GitNexus。
- 当问题同时包含“结构关系”与“精确定位”两类需求时，**必须混合使用 GitNexus 与 grep**，禁止只依赖单一手段得出结论。
- 所有通过 GitNexus 得出的结论，**最终都必须回到源码确认**；**禁止将图谱结果直接当作最终事实**。

### MCP 服务不可用规则

- 当发现 GitNexus MCP 服务不可用、无响应、报错，或当前环境未提供相关能力时，**必须直接明确告知用户 GitNexus MCP 服务不可用**。
- 出现上述情况时，**禁止伪造、猜测或假设任何图谱查询结果**。
- 当 MCP 不可用但任务仍可继续时，**必须立即切换到 grep、源码阅读、路径搜索等普通方式继续处理**。
- 当任务强依赖 GitNexus 的结构化能力而无法继续时，**必须明确说明受限点**，不得假装已完成结构化分析。
- 除非用户明确追问，否则**禁止主动输出索引建立命令、运维步骤或环境修复方案**。

### 代码检索规则

- 当需要检索代码时，**必须先判断当前问题属于“结构关系问题”还是“精确文本问题”**，再选择工具。
- 当问题是“谁依赖它、它依赖谁、谁调用了它、它属于哪个流程、改了它会影响什么”时，**必须优先使用 GitNexus**。
- 当问题是“这个字符串/字段/枚举/常量/配置键出现在哪、这段实现写在哪”时，**必须优先使用 grep / 文本搜索**。
- 当符号尚不明确时，**可以先做概念查找或文本定位**；一旦已确认具体符号，**必须立即切换到针对符号的结构化查询**。
- 当查询结果存在歧义、结果过散或覆盖不全时，**必须继续补充其他检索手段**，禁止基于单次检索草率下结论。

### `context` 使用规则

- 当已经知道具体类名、函数名或变量名，并且需要理解其上下游关系时，**必须优先使用 `context`**。
- 当查询目标存在同名方法、常见方法名或多处定义时，**必须结合 `file_path` 或 `uid` 做消歧**。
- 当问题关注类的继承关系、方法列表、被谁依赖、依赖了什么时，**应当优先依赖 `context` 的结果组织分析路径**。
- 当目标是枚举常量、属性访问型引用或纯文本出现位置时，**禁止把 `context` 当作主查询方式**。

### `impact` 使用规则

- 当准备修改类、Handler、Resource 或关键公共方法时，**在开始改代码前必须先执行 `impact` 分析影响范围**。
- 当使用 `impact` 时，**必须重点检查 `risk`、`byDepth`、`affected_processes`、`affected_modules`**。
- 当目标属于枚举常量、属性访问型引用或方法级覆盖不稳定场景时，**禁止只依赖 `impact` 结果**；**必须补 grep 或源码确认**。
- 当 `impact` 返回结果明显过少、缺失异常或与源码认知不一致时，**必须将其视为能力边界或覆盖不全，而不是直接认定“没有影响”**。

### `cypher` 使用规则

- 当 `context` 或 `impact` 无法满足需求时，**才应当使用 `cypher`**。
- 当需要追踪方法级调用链、内部调用路径或特定关系类型时，**必须优先考虑 `cypher`**。
- 对于简单的符号定位或普通关系查看，**禁止默认直接使用 `cypher` 作为入口工具**。
- 使用 `cypher` 得出的结构结论后，**仍必须回到源码确认关键调用细节**。

### `query` 使用规则

- 当只知道一个业务主题、功能概念或模糊问题，而尚不知道具体类名或函数名时，**才可以使用 `query`**。
- 当已经知道明确的类名、函数名或文件位置时，**禁止继续使用 `query` 作为主入口**，应直接转向 `context` 或 grep。
- 当 `query` 结果较散、噪音较多或概念混杂时，**必须先收敛出关键符号，再继续后续分析**。

### `detect_changes` 使用规则

- 当需要在提交前检查本次改动的影响面时，**应当优先执行 `detect_changes`**。
- 使用 `detect_changes` 后，**必须优先关注 `risk_level` 与 `changed_symbols`**。
- 当发现高风险符号时，**必须继续补 `impact` 分析**，禁止停留在提交级粗粒度结果。

### `rename` 使用规则

- 当需要做跨文件重命名时，**必须遵循固定顺序：先 `impact`，再 `rename(dry_run=true)`，复核后再正式执行**。
- 在未做 dry run 预览前，**禁止直接正式执行重命名**。
- 当 dry run 中存在低置信度编辑、异常改动或超出预期的改名范围时，**必须先复核再继续**。
- 重命名完成后，如环境支持，**应当确保索引或上下文已更新**。

### `api_impact` 使用规则

- 当准备修改 API handler、对外接口逻辑或路由处理器时，**应当优先执行 `api_impact` 或等价影响评估手段**。
- 若 `api_impact` 不可用或结果不足，**必须退回 `impact` + grep + 源码确认的组合方式**。

### 低优先级能力规则

- 在 `bk-monitor` 这类 DRF Resource 后端仓库中，`route_map`、`shape_check`、`tool_map` **不得作为主工具依赖** [[memory:dsmrltxt]]。
- 当存在 `context`、`impact`、`cypher`、grep 可用时，**必须优先使用这些更稳定的手段** [[memory:dsmrltxt]]。
- 如果上述低优先级能力返回空结果或弱相关结果，**不得据此推导“没有关系”或“没有影响”**。

### 修改前规则

- 当准备修改任何关键类、关键函数、公共接口或资源类时，**必须先完成“定位目标 → 结构理解 → 影响评估 → 源码确认”四步**。
- 当结构分析结果与源码实现不一致时，**必须以源码为准**。
- 当影响范围尚不清楚时，**禁止直接开始改代码**。

### 提交前规则

- 当准备提交修改时，**应当先检查当前改动影响面**。
- 当发现高风险改动、跨模块影响或公共能力变更时，**必须补充影响分析**。
- 当提交前仍存在“引用是否完整、影响是否遗漏、调用链是否断裂”的疑问时，**必须继续检索和确认，不得仓促结束分析**。

### 已知限制规则

- 必须默认认为：**方法级查询稳定性通常弱于类级查询**。
- 必须默认认为：**属性访问、枚举常量、字符串引用的覆盖并不完整**。
- 必须默认认为：**图谱结果只能辅助理解，不能替代源码事实**。
- 当结果异常、缺失或与预期不一致时，**应先怀疑能力边界、覆盖问题或环境状态，而不是立即下否定结论**。

### 最小可嵌入规则集

如需进一步压缩，可只保留以下强制规则：

- 当需要理解结构关系或评估影响时，**必须优先考虑 GitNexus**。
- 当需要精确定位字符串、字段、枚举、常量、配置或具体实现时，**必须优先使用 grep**。
- 当已知具体符号时，**必须优先使用 `context`**。
- 当准备修改关键目标时，**必须先执行 `impact`**。
- 当需要方法调用链时，**应优先补 `cypher`**。
- 当符号未知时，**才可以使用 `query`**。
- 当提交前检查风险时，**应优先使用 `detect_changes`**。
- 当执行重命名时，**必须先 dry run，再正式执行**。
- 当 MCP 不可用时，**必须直接告知用户，并改用普通搜索方式继续处理**。
- **所有结论最终必须以源码为准**。

### 一句话强制规则

**当需要结构关系时，必须优先考虑 GitNexus；当需要精确匹配时，必须优先使用 grep；当 MCP 不可用时，必须直接告知用户并退回普通检索；所有结论最终必须以源码为准。**
